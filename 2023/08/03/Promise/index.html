什么是异步

先来看下面这样一个关于时间规划的小学数学题：

题目：四年级小敏每天起床后要做的事情有：穿衣（3分钟），整理床铺（2分钟），洗脸梳头（8分钟），上厕所（5分钟），烧饭（20分钟），吃早饭（12分钟），完成这些工作要50分钟。你认为最合理的安排是多少分钟？

答案：
6:00 起床
├─ 6:00-6:03 穿衣（3分钟）
├─ 6:03-6:05 整理床铺（2分钟）
├─ 6:05-6:25 烧饭（20分钟）
│ ├─ 6:05-6:13 洗脸梳头（8分钟） ← 和烧饭完全重叠
│ ╰─ 6:13-6:18 上厕所（5分钟） ← 也在烧饭期间完成
╰─ 6:25-6:30 吃早饭（5分钟）

题目很简单，我们也能很快的想到答案，烧饭是一件很耗时且不需要太多人工参与的事情，我们就可以利用这段时间去处理其他事情。但这和异步有什么关系呢？

可以简单的说，像题目中这样：在同一时间段内，可以同时处理多件事情，不需要等前一个事情处理完成，才能处理下一个的场景，就是异步。在同一时间段内，只能处理一件事情，需要等待前一个事情处理完成后，才能开始处理下一个事情的场景，就是同步。

在程序的世界里，也存在许多类似于烧饭的处理，例如：文件的I/O处理、数据库操作等。在同步代码中，当一个任务（如读取文件、数据库查询）执行时，整个线程会被阻塞，无法处理其他请求，导致CPU空闲等待。这不仅会影响到用户的体验，还造成资源的浪费。

而JavaScript 是单线程语言，同一时间只能执行一个任务，那该如何解决这个问题呢？

回调函数

但如果使用回调函数来描述整个场景呢？

可以看到，函数嵌套的过深，不仅影响了代码的可读性，还会难以调试和维护。在JavaScript异步编程中，这种多层嵌套回调函数形成的复杂代码结构被称为回调地狱（Callback Hell）。

回调地狱（Callback Hell），也称为金字塔厄运（Pyramid of Doom），是指 JavaScript 中由于多层嵌套回调函数导致的代码结构复杂、难以维护的现象。

带来的问题：

​ ① 代码可读性差

​ ② 错误处理复杂

​ ③ 调试困难

​ ……

但在实际的开发中，我们不可避免的会遇到这种复杂的业务场景，该怎么解决回调地狱的问题呢？

Promise

JavaScript中的一种异步编程模式，解决了回调地狱的问题，帮助开发者更清晰地处理异步操作，使代码更易读、易维护。

核心原理与工作机制

Promise 本质上是一个内置构造函数，用于创建 Promise 对象实例。每个 Promise 实例内部维护着两个关键属性：

  1. PromiseState（状态机）：

    • pending：初始状态，表示异步操作正在进行中
    • fulfilled：操作成功完成（通过 resolve() 触发）
    • rejected：操作失败或被拒绝（通过 reject() 或抛出异常触发）

  2. PromiseResult（结果容器）：

    • 存储异步操作的返回值（resolve 传入的值）
    • 或存储错误信息（reject 传入的错误对象）

状态转换规则：

  • 创建时自动初始化为 pending 状态
  • 通过 resolve(value) 转换为 fulfilled，并存储结果值
  • 通过 reject(reason) 或抛出异常转换为 rejected，并存储错误信息
  • 状态转换是单向且不可逆的，一旦确定就不可更改

执行流程：

  1. 实例化 Promise 时，执行器函数（executor）立即执行
  2. 通过 then() 方法注册回调：
    • 当状态变为 fulfilled，触发第一个回调函数
    • 当状态变为 rejected，触发第二个回调函数（或 catch()）
  3. 通过 finally() 注册最终回调（无论成功失败都会执行）

优点

  • 避免回调地狱
  • 链式调用
  • 错误处理更简单
  • …..

基本用法

核心特点

特性 说明
状态不可逆 pending → fulfilled（成功）或 rejected（失败）
链式调用 .then() 返回新的 Promise，可继续 .then()
错误冒泡 错误会一直传递到最近的 .catch()
微任务队列 Promise 回调属于微任务（比 setTimeout 宏任务优先级高）

常用方法

以下是 Promise 静态方法的对比汇总表格：

方法 描述 成功条件 失败条件 返回值 特点
Promise.resolve() 创建立即完成的Promise 总是成功 无 传入的值 快速包装值为Promise
Promise.reject() 创建立即拒绝的Promise 无 总是失败 传入的错误 快速创建拒绝的Promise

{status: 'fulfilled', value}

Promise.race() 处理多个Promise 第一个完成的Promise（成功/失败） 第一个失败的Promise 第一个完成的结果 只关心最先完成的结果
Promise.any() 处理多个Promise 任意一个Promise成功 全部Promise失败 第一个成功的结果 忽略拒绝，只取第一个成功值

async/await

是什么？

async/await 是 ES2017 引入的语法糖，基于 Promise 的异步编程解决方案，让异步代码拥有同步代码的书写方式。

核心本质：

  • async：将函数标记为异步，总是返回 Promise 对象
  • await：暂停异步函数执行，等待 Promise 解决

用法

1.使用范围

await只能在async函数中使用，或ES模块的顶层作用域中使用。await阻塞的只是异步函数内部的代码，不会影响外部代码。

ES模块的顶层作用域是什么？

ES模块的顶层作用域指的是在模块的最外层，不在任何函数或类内部的代码块。这种情况下，顶层作用域的代码会在模块加载时立即执行。

重构时间规划问题

JS是单线程的，那么它是如何实现异步处理的呢？

在深入探讨解决方案之前，让我们先理解一个概念：事件循环（Event Loop）。JavaScript 通过事件循环（Event Loop）机制，在保持单线程执行模型的同时，实现了异步操作的高效处理。下面我们将详细解析这一重要机制。

原理：事件循环（Event Loop）

事件循环是JavaScript中的一个的运行时模型，负责程序执行期间的资源管理、任务处理和协调操作，它定义了代码如何被调度和执行的。

关键组件

组件 说明 常见例子

函数调用
任务队列（Task Queue） 即宏任务队列，是事件循环中处理”常规”异步任务的队列。 setTimeout、

执行流程

简化的执行顺序：1. 同步代码 → 2. 微任务 → 3. 宏任务

  1. 调用栈检查阶段
    • 事件循环首先检查调用栈(Call Stack)状态
    • 若调用栈不为空，则继续执行栈中的同步代码
    • 若调用栈为空，则进入任务队列处理阶段
  2. 微任务处理阶段
    • 当调用栈清空后，事件循环会优先处理微任务队列(Microtask Queue)
    • 依次执行微任务队列中的所有任务，直到队列完全清空
  3. 宏任务处理阶段
    • 在微任务队列清空后，从宏任务队列(Macrotask Queue)中取出一个任务执行
    • 执行完毕后立即返回微任务检查阶段

上述过程形成持续的事件循环(Event Loop)，每次宏任务执行后都会重新检查微任务队列，这种机制保证了高优先级任务的及时处理。
